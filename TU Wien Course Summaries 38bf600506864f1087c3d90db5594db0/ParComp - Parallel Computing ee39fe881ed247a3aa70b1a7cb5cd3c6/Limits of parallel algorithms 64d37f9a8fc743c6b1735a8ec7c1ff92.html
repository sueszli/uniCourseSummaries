Limit of absolute speedup

Linear speedup Sp(n)∈O(p)\small S_p(n) \in O(p)Sp​(n)∈O(p)﻿ can not be exceeded.

Tseq(n)≤p⋅Tpar(p,n)\color{pink}\small \small \text{Tseq}(n) \leq p \cdot \text{Tpar}(p,n)Tseq(n)≤p⋅Tpar(p,n)﻿

Perfect speed-up is rare and hardly achievable.

Sometimes we experimentally get super-linear speedup but its because of randomization, non determinism or because algorithms are not comparable (also possible by using caches on multiple processors).

    Sequential simulation construction

    (Wrongly) assume that we can simulate a parallel algorithm sequentially by:

    1. executing one step of each process, for C(n)\small C(n)C(n)﻿ iterations
    1. all steps of a single process until a communication/synchronization point, then the steps of the next process and so on, ...

    Proof through contradiction

    Assume in Tsim(n)\small \text{Tsim}(n)Tsim(n)﻿ simulates Tpar(p,n)\small \text{Tpar}(p,n)Tpar(p,n)﻿

    p⋅Tpar(p,n)≥Tsim(n)\small p \cdot \text{Tpar}(p,n) \geq \text{Tsim}(n)p⋅Tpar(p,n)≥Tsim(n)﻿

  • Proof

    Lets say there is a work-optimal algorithm with:

    Tpar(p,n)=max⁡0≤i<pTi(n)\small \text{Tpar}(p,n) = \max_{0\leq i<p} T_i(n)Tpar(p,n)=max0≤i<p​Ti​(n)﻿

    ∑0≤i<pTi(n)=Tseq(n)\small \sum_{0\leq i <p} T_i(n) = \text{Tseq}(n)∑0≤i<p​Ti​(n)=Tseq(n)﻿

    We schedule Par\small \text{Par}Par﻿ on a smaller number of processors p′\small p' p′﻿ in such a way that no processor has any “idle time” (pretty difficult to do this in practice).

    Then:

    ∑0≤i<p′Ti(n)=p′⋅Tpar(p′,n)∈O(Tseq(n))\small \sum_{0\leq i <p'}T_i(n) = p' \cdot \text{Tpar}(p',n)\in O(\text{Tseq}(n))∑0≤i<p′​Ti​(n)=p′⋅Tpar(p′,n)∈O(Tseq(n))﻿

    This way the algorithm is cost-optimal and therefore has linear speedup.

Overhead

Parallelization Overhead

= Work that does not have to be done by a sequential algorithm like cooridination (communication, synchronization) and redundant computation.

When there is no overhead we can parallelize perfectly: Tpar(p,n)=Tseq(n)/p\small \text{Tpar}(p,n) = \text{Tseq}(n)/pTpar(p,n)=Tseq(n)/p﻿

Differece betweeen max and min work+overhead.

Load balancing

Achieving an even amount of work for all processors i,j\small i,ji,j﻿:

Tpar(i,n)≈Tpar(j,n)\color{pink}\small \text{Tpar}(i,n) \approx \text{Tpar}(j,n)Tpar(i,n)≈Tpar(j,n)﻿

Static, oblivious

Splitting problem in p\small pp﻿ pieces, regardless of the input (except it’s size).

Static, problem dependent, adaptive

Splitting problem in p\small pp﻿ pieces, using the structure of the input.

Dynamic

Dynamically (during program execution) readjusting the work assigned to processors.

This means some overhead.

Amdahl’s Law (parallel version): Sequential bottleneck / overhead

Let program of Seq\small \text{Seq}Seq﻿ contain two fractions, independent of n\small nn﻿:

  • “perfectly parallelizable” fraction r\color{pink}\small rr﻿
  • “purely sequential” fraction s=(1−r)\color{pink}\small s=(1-r)s=(1−r)﻿ (can’t be parallelized at all)

The maximum achievable speedup (independent of n\small nn﻿):

Sp(n)=1/s\color{pink}\small S_p(n) = 1/sSp​(n)=1/s﻿

Scalability

If an algorithm does not have constant efficiency and it’s speedup dependens on n\small nn﻿

we can aim to maintain some constant efficiency e\small ee﻿ through

increasing the problem size n\small nn﻿ with the number of processors p\small pp﻿

using the iso-efficiency function.

Weak scalability

Parallel algorithm scales weakly relative to sequential algorithm

if there is a slowly growing iso-function f\small ff﻿ such that:

n∈Ω(f(p))  ⟹  Ep(n)=e\color{pink}\small n \in \Omega(f(p)) \implies E_p(n) = en∈Ω(f(p))⟹Ep​(n)=e﻿

The growth of f\small f f﻿ is given by the input-size-scaling-function g\small gg﻿.

  • Average work per processor must be constant by increasing n\small nn﻿.

    w=Tseq(n)/p\small w = \text{Tseq}(n)/pw=Tseq(n)/p﻿

  • Tpar(p,n)\small \text{Tpar}(p,n)Tpar(p,n)﻿ must be constant

g(p)=1/Tseq(p⋅w)\color{pink} \small g(p) = 1/\text{Tseq}(p\cdot w)g(p)=1/Tseq(p⋅w)﻿

f(p)∈O(g(p))\color{pink}\small f(p) \in O(g(p))f(p)∈O(g(p))﻿

Exception to this rule:

If sequential time > linear then constant efficiency requires n\small nn﻿ to grow faster than said here.

In that case constant work is maintained with decreasing efficiency.

Strong scalability

Algorithm is strongly scalable if f(p)=O(1)\small f(p) = O(1)f(p)=O(1)﻿.

This is almost never the case: all algorithms are at best weakly scalable.

At least as much work is required as there are processors.

But often, constants and lower order terms can safely be ignored - then that the algorithm is strongly scalable for some range of n\small nn﻿ and p\small pp﻿.

Weak scaling analysis

Goal: running larger problem in same amount of time.

  • n\small nn﻿ is increased
  • Average work per processor stays fixed

The total problem size n\small nn﻿ is proportional to the number of processors used.

  • n\small nn﻿ is kept fixed
  • p\small p p﻿ is increased
