There are no dependencies between loop iterations ‚Üí Data parallel computation.

Lower work to O(‚à£active‚à£‚ãÖf)\small O(|\texttt{active}| \cdot f)O(‚à£active‚à£‚ãÖf)Ôªø by iterating only over active indices:

Blocking technique

Use when applicable (not always applicable).

Goal: reducing communication / synchronization steps.

  • Blocking technique (general view)
    1. Divide into sub-problems of size n/p\small n/pn/pÔªø that can be solved sequentially or in parallel.
    1. Assign subproblem to each processor.

      Step 1 and 2 should be fast: ie. O(1)\small O(1)O(1)Ôªø, O(log‚Å°n)\small O(\log n)O(logn)Ôªø, ...

    1. Solve subprolems using sequential algorithm.

      Should be perfectly parallelizable: ie. O(n/p)\small O(n/p)O(n/p)Ôªø

    1. Use parallel algorithm to combine subproblem results.

      Should be fast: ie. O(log‚Å°p)\small O(\log p)O(logp)Ôªø

      Total cost must be less than O(n/p)\small O(n/p)O(n/p)Ôªø

    1. Apply combined result to subproblem solutions.

      Should be perfectly parallelizable

Blocking technique (alternative view)

  1. Use work-optimal algorithm to shrink algorithm

    Make blocks of size Œò(n/p)\small \Theta(n/p)Œò(n/p)Ôªø using O(f(p,n))\small O(f(p,n))O(f(p,n))Ôªø time steps per processor.

  1. Use fast and possibly non work-optimal algorithm on shrunk problem

    Solve p\small ppÔªø subproblems in parallel.

  1. Unshrink, compute final solution with work-optimal algorithm

    Combine into final solution using O(g(p,n))\small O(g(p,n))O(g(p,n))Ôªø time steps per processor.

Usually we can get from O(n)\small O(n) O(n)Ôªø to O(n/log‚Å°n)\small O(n/ \log n)O(n/logn)Ôªø

Resulting parallel algorithm is cost-optimal if f(p,n)‚ààO(n/p)\small f(p,n)\in O(n/p)f(p,n)‚ààO(n/p)Ôªø and g(p,n)‚ààO(n/p)\small g(p,n) \in O(n/p)g(p,n)‚ààO(n/p)Ôªø.
