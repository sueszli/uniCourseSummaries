Definition

Communication protocol

rules for data transmission in a network. i.e. HTTP

Cryptographic protocol

communication protocol for encrypted messages.

used in: e-banking, e-commerce, file sharing, online messengers

Cryptography is not enough: the attacker can circumvent cryptography and break the protocol.

Possible flaws can be in the:

  • Protocol design / implementation
  • Encryption design

Automated Protocol Analysis

Protocols are complicated and attacks are difficult to find. We need Automatation, formal proofs.

Requirements:

  1. Specification language for the protocol (= process calculus: λ\lambdaλ﻿-Calculus, π\piπ﻿-Calculus)
  1. Formal definition of Security
  1. Automated security analysis (ProVerif)

Attacks

Interleaving Attack

Message from one protocol-session is used in other ones.

Reflection Attack

(Subtype of Interleaving attack)

Message sent back to the person that generated it (in a different session).

Problem: (If we are using symmetric encryption) the sender can not verify who sent the message.

Solution: Identifier (sender / receivers name)

Man-in-the-middle Attack

The victim BBB﻿ then thinks he is communicating with ◯\bigcirc◯﻿ but he is actually communicating with AAA﻿.

The attacker has access to nA,nBn_A,n_BnA​,nB​﻿ and can decrypt all messages from BBB﻿.

A←{B,nB}pkA−◯←{B,nB}pk◯−BA \larr \{B, n_B\}_{pk_A} - \bigcirc \larr \{B, n_B\}_{pk_\bigcirc} - BA←{B,nB​}pkA​​−◯←{B,nB​}pk◯​​−B﻿ Attacker reads Bobs messages, has his nBn_BnB​﻿.

A−{nB,nA}pkB→BA - \{n_B, n_A\}_{pk_B}\rarr BA−{nB​,nA​}pkB​​→B﻿

A←{nA}pkA−◯←{nA}pk◯−BA \larr \{n_A\}_{pk_A} - \bigcirc \larr \{n_A\}_{pk_\bigcirc} - BA←{nA​}pkA​​−◯←{nA​}pk◯​​−B﻿ Attacker gets nAn_A nA​﻿ from Bob.

Solution: Needham-Schroeder-Lowe Protocol

A←{B,nB}pkA−BA \larr \{B, n_B\}_{pk_A} - BA←{B,nB​}pkA​​−B﻿

A−{A,nB,nA}pkB→BA - \{\textcolor{pink}A,n_B, n_A\}_{pk_B} \rarr BA−{A,nB​,nA​}pkB​​→B﻿ (new identifier!)

A←{nA}pkA−BA \larr \{n_A\}_{pk_A} - BA←{nA​}pkA​​−B﻿

Then the attack is not possible because bob does not think he is talking to ◯\bigcirc◯﻿ anymore.

Basic Security Goals

There are many more properties, but we only focus on these.

Secrecy

Only authorized end-points should be able to read messages.

Secrecy is undecidable - infinite number of opponents OOO﻿. (automatized proofs with tools possible).

Protocol PPP﻿ preserves secercy of M⇔M \LrarrM⇔﻿

∀O, c∈fn(O): P∣O\forall O,~ c\in fn(O): ~P \mid O∀O, c∈fn(O): P∣O﻿ does not output MMM﻿ on channel ccc﻿.

Where channel ccc﻿ is a free name (public) known to process OOO﻿.

If the attacker can not output MMM﻿, it has no access to it.

Integrity

The recipient of a message should be able to determine changes during transmission.

Sender and receiver should agree on their roles.

Non-Injective Agreement

  • Identity: The recipient should be able to verify the requesters identity.
  • Order: In all execution traces, if we reach the endendend﻿ we must have also reached a beginbeginbegin﻿.

Formally

PPP﻿ guarantees noninjective agreement iff:

∀O:P∣O→∗ new aundefined.(event end(A,B,M)∣Q)  ⟹  \forall O: \quad P \mid O \rightarrow^{*} \text { new } \widetilde{a} .(\text {event } {end}(A, B, M) \mid Q) \implies∀O:P∣O→∗ new a.(event end(A,B,M)∣Q)⟹﻿

Q≡ event begin(A,B,M)∣Q′Q \equiv \text { event } {begin}(A, B, M) \mid Q^{\prime}Q≡ event begin(A,B,M)∣Q′﻿

  • In depth explaination

    PPP﻿ is the process we want to analyse that runs parallel to any opponent OOO﻿.

    QQQ﻿ is the process left after PPP﻿ and Q′Q'Q′﻿ is the process left after QQQ﻿.

    →∗\rarr^*→∗﻿ stands for n-step reductions.

    aundefined\widetilde aa﻿ stands for a sequence of variables that are bounded in the following process.

    This is a recursive definition:

    It means that in our closed process P PP﻿ for any opponent OO O﻿ that we run in parallel, after n-reductions and binding arbitrary many variables to our process (which we simply refer to as aundefined \widetilde a a﻿ - we must always reach two processes running in parallel from which both fulfill the non-injective agreement.

    Because these processes can run concurrently we can have any arbitrary ordering as long as an end event always follows after a begin event.

    Therefore these orderings would both be valid although the required end event does not follow immediately after the begin event:

    Begin1 →\rightarrow →﻿ Begin2 →\rightarrow →﻿ End1 →\rightarrow →﻿ End2

    Begin1 →\rightarrow→﻿ Begin2 →\rightarrow →﻿ End2 → \rightarrow→﻿ End1

ProVerif

Cryptographic protocol verifier based on Horn clause resolution.

Input notation: applied pi-calculus.

Outcomes

Possible outcomes after modelling a protocol:

a) proven security

b) found attacks - (could be a false positve because of abstraction)

c) not finding anything / not terminating

Translation

Is what ProVerif does:

our source-code →\rarr→﻿ applied pi-calculus →\rarr→﻿ logical formulas (horn-clauses)

The horn clauses are then sent to a theorem prover.

The translations are sound (= error free) but incomplete:

There can be no false-negatives of security, only false-positives.
