Information Flow (compact)

mathematical model of the system (of the code) → code analysis → (ideally automatic) proof techniques for security

Goal:

  • controlling information flow
  • not being too restrictive ("declassifying" some information when necessary)

Side channels

Mechanisms for getting sensitive information through the observable behaviour of a computing system.

Explicit flow

Direct assignment.

l := h

Implicit flow

Leak information through the control structure of the program.

h := h mod 2;
l := 0;
if h = 1
	then l := 1 //equivalent to l := h mod 2
	else skip

Termination channels

Leak information through termination / non-termination.

while h=1
	do skip //does not terminate if h = 1

Timing channels

Leak of information through the total execution time of program / single instruction.

Can be subtler and in example check cache access time.

if h=1
	then C_long //computation that takes long
	else skip

Probabilistic channels

Leak information by changing the probability distribution of observable data. (different stochastic properties)

Example: multi-thread program where secret is a number between 1 and 100 and random(100) randomly yields a number between 1 and 100.

leak := secret; || leak := random(100);

The behaviour of the execution scheduler can be observed.

Power channels

Power consumption by computer

Resource exhaustion channels

exhaustion of a finite, shared resource, such as a memory or a disk space

Non-Interference = formal definition of confidentiality

Note: This definition is incomplete:

  • termination-insensitive
  • timing-insensitive (running time data, cache data)
  • does not work with multi-threading
  • does not work with non-determinism

Security Type System

Non-interference definition contains a universal quantification over all possible inputs → undecidable. - we need static analysis = security type system.

Considering termination

Either both memories must not terminate, or if they do terminate, then they must have the same lower part.

Considering timing

Issue: Branches taking different times

Solution: Transforming out timing leaks

Cross-copy low slices = dummy instructions (like skip sequences) in each branch that so that both take an equal amount of time.

Change of typing rules

Difference 1: number of branching instructions must be equal c1∼c2c_1 \sim c_2c1​∼c2​﻿ to prevent timing channels.

Difference 2: The guard must be low and the entire instruction must be checked under a low pcpcpc﻿.

We don't want different looping times based on the guard.
